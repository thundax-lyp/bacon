# HOW TO RUN TODO COLLABORATION

## 1. Purpose

本文档说明：AI、人类和 `TODO.md` 如何协作完成任务记录、拆解、审阅、执行和关闭。

目标是让 `TODO.md` 既保持任务队列的清晰，又能承载人机达成共识前的任务入口。

## 2. Scope

当前范围：

- AI 将拆解任务、执行任务或待讨论项写入 `TODO.md`
- 人类阅读并与 AI 讨论 `TODO.md` 项
- AI 将达成共识的拆解任务拆解为执行任务
- 人类审阅拆解后的 TODO
- AI 执行并在完成后删除 TODO

不在范围内：

- 不替代具体业务需求文档
- 不替代架构红线或文档规则
- 不记录已完成任务历史
- 不把低置信度观察项写入 `TODO.md`

## 3. Bounded Context

`TODO.md` 是未关闭任务面板。

在 AI 协作场景中，`TODO.md` 承载三类未关闭任务：

- 拆解任务：目标已经确定，必须按指定 Runbook 拆成执行任务
- 执行任务：已经达成共识，可以进入实现、测试、文档同步和提交
- 待讨论项：需要人类先做决策，不能直接执行

拆解任务和待讨论项不是执行许可。拆解并经人类审阅后的执行任务，才是执行入口。

## 4. When To Use

- AI 通过扫描、讨论或复盘发现一个值得推进的任务
- 人类希望先围绕任务目标、边界和优先级形成共识
- 当前任务需要先拆解，再进入实现
- 当前任务可能影响架构、文档、测试或多个模块

## 5. Do Not Use For

- 已经非常明确的一步式小改动
- 低置信度、暂时无法行动的观察项
- 纯讨论材料、灵感记录或完成历史
- 尚未确认是否属于本仓库范围的想法

## 6. Pre-Checks

写入 TODO 前，AI 固定先判断：

1. 该任务是否已经足够重要，值得打扰人类阅读。
2. 该任务是否能用一句话说明问题或目标。
3. 该任务是否能明确归类为拆解任务、执行任务或待讨论项。
4. 该任务是否不是低置信度观察项。
5. 该任务是否存在明确的下一步动作、拆解依据或决策问题。

只要以上任一问题不能明确回答，就不要写入 `TODO.md`。

## 7. Steps

### 7.1 Add A TODO Item

AI 可以将拆解任务、执行任务或待讨论项写入 `TODO.md`。

TODO 项必须包含：

- 任务目标
- 任务类型
- 范围对象、处理动作、验收点或决策要求

拆解任务不得包含过细实现步骤，不得伪装成已经达成共识的执行任务。
待讨论项不得包含实现步骤，不得伪装成执行任务。

### 7.2 Discuss The TODO Item

人类阅读 TODO 项后，与 AI 讨论以下内容：

- 任务是否值得做
- 边界是否正确
- 优先级是否正确
- 验收口径是否清楚
- 是否需要补充文档、测试或治理规则

讨论完成前，AI 不得把拆解任务或待讨论项当作执行许可。

### 7.3 Split Into Executable TODO

达成共识后，AI 将拆解任务按指定 Runbook 拆解为执行任务。

拆解后的 TODO 必须满足：

- 每项都有明确动作
- 每项可以独立判断完成或未完成
- 每项边界清楚，不混合不相关判断
- 每项能对应到代码、测试、文档或提交动作
- 每项引用对应 Runbook 步骤或依据文档

### 7.4 Review The Split TODO

人类审阅拆解后的 TODO。

审阅重点：

- 任务粒度是否合适
- 执行顺序是否合理
- 是否遗漏验收条件
- 是否把讨论结论拆偏
- 是否引入了不必要的扩展

审阅通过后，AI 才进入执行。

### 7.5 Execute And Close

AI 按确认后的 TODO 执行。

执行完成后：

- 已完成 TODO 必须删除
- 未完成部分必须拆分或收窄为剩余任务
- 完成历史保留在 commit 或 PR 中
- 删除 TODO 应与对应代码、测试、文档修改放在同一闭环提交中

## 8. TODO Item Format

### 8.1 Split Task

拆解任务用于把宏观目标按 Runbook 拆成多条执行任务，防止后续执行发生语义漂移。

格式：

```markdown
- [ ] `主 scope`：按 Runbook 拆解某某任务
  - 任务类型：拆解任务
  - 依据文档：`docs/.../XXX-RUNBOOK.md`
  - 原始目标：...
  - 拆解要求：...
  - 输出要求：...
  - 审阅要求：...
  - 验收点：...
  - 重要度：N/10
```

约束：

- `主 scope` 只能有一个，表示拆解任务的责任归属
- `依据文档` 必须指向稳定 Runbook 或治理文档
- `拆解要求` 必须明确按哪些步骤拆，不得合并跨层步骤
- `输出要求` 必须说明生成的执行任务格式和引用关系
- 拆解完成后，删除本拆解任务，只保留经人类确认的执行任务

### 8.2 Executable Task

执行任务用于直接实现、测试、关闭和提交。

格式：

```markdown
- [ ] `主 scope`：任务标题
  - 任务类型：执行任务
  - 依据文档：`docs/.../XXX.md`
  - 关联范围：`scope` / `scope`
  - 范围对象：...
  - 处理动作：...
  - 验收点：...
  - 重要度：N/10
```

约束：

- `主 scope` 只能有一个，表示任务所有权
- `关联范围` 可省略，只表示影响面或协作对象，不扩大执行所有权
- 一个执行任务只表达一个主动作
- `范围对象`、`处理动作`、`验收点` 必填
- 如果 `处理动作` 包含多个互相独立的动作，必须拆分
- 如果 `验收点` 需要跨多个无关能力判断，必须拆分或收窄

### 8.3 Discussion Item

待讨论项用于记录需要人工决策的问题，不得直接执行。

格式：

```markdown
- [ ] 决策问题
  - 任务类型：待讨论项
  - 关联任务：...
  - 决策要求：...
  - 重要度：N/10
```

约束：

- 待讨论项必须放在 `## 待讨论项`
- `决策要求` 必须说明需要人类固定什么口径
- 决策完成后必须删除待讨论项
- 如果决策后仍需执行，新增明确的拆解任务或执行任务

## 9. Files To Touch

本流程通常涉及：

- 根目录 `TODO.md`
- 与任务相关的业务代码、测试或文档
- 收口时可能涉及的 commit

本流程通常不要求新增治理文档。只有当讨论结论形成稳定规则时，才同步治理文档。

## 10. Common Mistakes

- AI 将拆解任务或待讨论项直接当作执行许可
- AI 把低置信度观察项写入 `TODO.md`
- 人类尚未确认边界，AI 就开始拆解或实现
- 拆解后的 TODO 粒度过细，制造噪音
- 拆解后的 TODO 混入未经讨论的新目标
- 执行任务标题包含多个主 scope，导致任务所有权不清
- 完成后在 `TODO.md` 保留打勾历史

## 11. Verification

进入执行前固定自检：

1. TODO 项是否已经明确归类为拆解任务、执行任务或待讨论项。
2. 拆解后的 TODO 是否经过人类审阅。
3. 每个执行 TODO 是否能独立验收。
4. TODO 中是否没有完成历史和低置信度观察项。
5. AI 是否只执行已经确认的执行任务。

## 12. Commit Guidance

如果本流程只修改 `TODO.md`，提交信息应说明任务协作状态变化。

如果本流程完成了代码、测试或文档改动，应按对应任务闭环提交，并在同一提交中删除或收窄已完成 TODO。
