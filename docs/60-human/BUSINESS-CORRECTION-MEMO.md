# 如何纠正判断失误：一份未完成的备忘录

本文不是治理规则，也不是 `HOW-TO`。

它只记录一次关于“AI 执行大量任务后，作者如何纠正历史判断偏差”的讨论。这里面的概念还没有固定，不应被 AI 当作默认执行规范。

---

## 问题从哪里来

普通的代码回退处理的是一个很小的问题：当前任务失败了，把当前任务撤掉即可。

但作者真正担心的不是这个场景。

更复杂的场景是：AI 已经连续执行了一大批任务，作者休假回来后观察这些任务，发现其中某几个任务存在设计偏差。偏差并不一定表现为编译失败，也不一定表现为测试失败。它可能只是一个业务判断错了，例如状态机边界、幂等键设计、Facade 契约、租户上下文归属、权限缓存失效语义。

这个时候不能简单 `revert` 某个 commit。

因为后续任务可能已经沿着这个判断继续生长，有些后续成果是错的，有些后续成果仍然是对的。如果直接回退起点 commit，可能会摧毁很多正确成果；如果只在当前代码上局部 patch，又可能漏掉沿着错误判断扩散出去的影响。

这里需要处理的不是“代码版本回退”，而是“业务判断纠偏”。

## 石子和涟漪

这件事更像一颗石子扔进湖里。

错误判断是石子落水的位置。后续任务是从这个位置向外扩散的涟漪。涟漪会碰到一些区域，但不会天然覆盖整个湖面。

作者真正需要的是：

- 标定石子落点，也就是错误判断的起点。
- 看清涟漪的扩散方向，也就是哪些后续任务依赖了这个判断。
- 画出收敛范围，也就是这次纠偏应该改到哪里、检查到哪里、停在哪里。
- 证明边界之外不需要动，而不是因为没有检查才没动。

这个问题的关键，不是“能不能回退”，而是“如何知道回退或纠偏应该停在哪里”。

## 它像 Codex 的收敛过程

Codex 本身也在做收敛。

它修改代码，运行 compile/test，看到失败，再修改，直到机械验收通过。它的终点相对清楚：编译通过、测试通过、lint 通过。

作者要做的是更高一层的收敛。

它不是以 compile/test 为终点，而是以业务判断重新稳定为终点。问题是业务没有天然的 `mvn test`。一个设计判断是否稳定，经常要靠作者观察、业务不变量、跨域契约、数据语义、运行处置共同确认。

所以这里需要某种“业务编译器”。

它不一定是工具，也不一定一开始就是自动化系统。它可能先是一份纠偏闭环记录，用来回答：

- 旧判断是什么。
- 新判断是什么。
- 哪些任务直接依赖旧判断。
- 哪些任务只是间接受影响。
- 哪些边界明确不受影响。
- 哪些业务不变量证明新判断已经站住。

## 可能的收敛标准

讨论中出现过一个暂时有用的短句：

> 单一判断、影响闭合、不变量可验、边界可解释。

它还不是正式规则，但可以作为后续继续思考的锚点。

“单一判断”指同一个业务问题最后只能剩一个有效版本。文档、代码、测试、契约不能各说一套。

“影响闭合”指从错误判断出发，直接依赖已经处理，间接依赖已经检查。不能因为影响链太长就假装它不存在。

“不变量可验”指纠偏最后必须落到业务不变量上。例如订单终态不能反向迁移，支付成功事实不能被订单关闭覆盖，租户上下文不能跨请求污染，权限变更后不能读取过期授权。

“边界可解释”指涟漪停在哪里必须有理由。比如外部 HTTP 契约未变，某个 Facade 未变，存量数据不受影响，或者某个域只读了结果但没有依赖旧判断。

## 不要急着做成规则

现在还不应该急着把它写成 `HOW-TO`。

原因是这个问题还没有完全想清楚。它涉及任务树、提交历史、业务判断、后续任务依赖、作者审阅和 harness 自动纠偏。太早规则化，容易把一个还在生长的想法写死。

比较合适的下一步，是先把这种纠偏记录当作人类备忘录保存。等真实出现 2-3 次类似场景，再观察哪些字段反复出现，哪些判断总是需要作者确认，哪些边界最容易漏。

到那时，再考虑是否需要形成 `Business Correction Closure`。

## 一个尚未定型的轮廓

如果未来要继续推进，它可能不是普通 rollback，也不是新的版本管理器起点，而是一套围绕“设计判断”的记录系统。

它可能记录：

- `wrongDecision`：错误判断。
- `newDecision`：新判断。
- `rootTask`：偏差起点任务。
- `impactRings`：影响圈。
- `boundaryStopReason`：收敛边界为什么停在这里。
- `businessInvariants`：被重新确认的不变量。
- `contractImpact`：外部契约是否变化。
- `dataRuntimeImpact`：存量数据、消息、幂等记录、部署和回滚是否受影响。
- `closureProof`：文档、测试、人工确认和 commit 证据。

这些字段现在只是草图。

真正重要的是背后的判断：作者不是要把湖水倒回去，而是要找到石子，画出涟漪半径，然后只重塑这一圈水面。

## 留给后面的自己

如果以后再看这份备忘录，可以先问几个问题：

- Bacon 的任务树是否已经能表达“后续任务依赖了哪个业务判断”。
- commit 历史是否足够表达判断顺序，还是需要额外 task journal。
- TODO 项是否应该记录 `dependsOn`、`exports`、`touchedSurfaces` 这类字段。
- 业务不变量是否能成为比 compile/test 更高一层的收敛终点。
- 纠偏完成后，是否需要留下一份可回看的 `Business Correction Closure`。

这些问题还没有答案。

但方向已经比较清楚：AI 可以执行很多任务，作者真正要守住的是判断。纠正判断失误时，系统需要帮助作者看见影响，而不是只提供一个粗暴的回退按钮。
