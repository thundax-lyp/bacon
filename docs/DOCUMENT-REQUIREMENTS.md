# DOCUMENT REQUIREMENTS

## 1. Purpose

本文档用于定义项目内需求文档和 AI 输入文档的统一写作要求。  
目标是让文档适合 AI 读取、适合工程实现、适合后续持续维护。

## 2. File Naming

- 文档文件名使用大写英文
- 文件名可使用 `-` 连接单词
- 文件名不得使用中文
- 文件名不得使用空格

## 3. Language Rules

- 文档说明内容使用中文
- 代码定义相关名称使用英文
- 下列内容必须使用英文原文：
  - 模块名
  - 类名
  - 接口名
  - 服务名
  - `DTO` / `VO` / `Facade` / `Repository` / `Mapper`
  - 字段名
  - 枚举值
  - 缓存键
  - 权限编码
  - 登录方式类型
  - 资源类型
- 不为了“纯中文”而翻译代码概念
- 不为了“纯英文”而把业务说明改成英文

## 4. Content Principles

- 文档必须清晰、明确、可执行
- 文档必须适合 AI 读取
- 文档必须适合直接指导实现
- 文档不得包含冗余说明
- 文档不得过度简化
- 文档不得保留模糊口径
- 文档不得出现同一规则在多处重复且表述不一致

## 5. Requirement Style

- 使用确定性表达
- 不使用“建议”“可考虑”“视情况”“后续再看”“如有需要”等不确定措辞
- 已确认的规则必须直接写成约束
- 需要固定的内容必须明确写成“固定”
- 不存在待定时，明确写 `无`

## 6. Structure Requirements

文档优先采用以下结构：

1. `Purpose`
2. `Scope`
3. `Bounded Context`
4. `Module Mapping`
5. `Core Domain Objects`
6. `Global Constraints`
7. `Functional Requirements`
8. `Key Flows`
9. `Non-Functional Requirements`
10. `Open Items`

要求：

- 先写全局边界，再写对象能力
- 先写统一规则，再写功能需求
- 先写稳定约束，再写补充说明
- 同类规则集中在一个位置，不分散重复

## 7. Scope Writing Rules

- 当前范围与不在范围内必须明确分开
- 当前范围内全部功能如果同批交付，必须明确写明
- 如果不做分期交付，不写优先级章节
- 如果存在范围外能力，必须明确列出
- 范围内项目必须在后续正文中有对应定义
- 范围中出现的能力，必须能在功能需求或规则中找到落点

## 8. Boundary Writing Rules

- 明确模块归属
- 明确职责边界
- 明确谁拥有主数据
- 明确谁负责流程
- 明确谁只能读、谁可以写
- 跨域调用必须说明依赖层级
- 不允许出现一个能力同时归属两个模块但不说明边界

## 9. Naming Rules

- 统一术语
- 同一概念在文档中只使用一种主名称
- 文中首次出现的重要对象、接口、服务、枚举应使用固定英文名称
- 业务规则使用中文描述
- 命名格式需要固定时，必须写明格式

例如：

- `permission code` 格式必须写成 `domain:model:action`
- 缓存键必须写出完整格式
- 枚举值必须逐项列出

## 10. Rule Writing Rules

- 全局规则写在 `Global Constraints`
- 对象特有规则写在对应功能章节
- 非功能规则单独写在 `Non-Functional Requirements`
- 会影响表结构、唯一索引、缓存、权限、认证、日志的规则必须写明
- 所有唯一性规则必须明确是“全局唯一”还是其他范围
- 所有缓存规则必须明确键、内容、失效触发
- 所有认证相关规则必须明确归属和会话失效行为

## 11. Functional Requirement Rules

每个功能章节至少包含：

- 功能对象
- 功能能力
- 必要补充约束

如果某个功能涉及以下内容，必须显式写出：

- 唯一性约束
- 启用/停用行为
- 删除策略
- 可见性规则
- 绑定关系
- 查询范围
- 鉴权规则
- 缓存影响

## 12. Data And Model Rules

- 核心领域对象必须单独列出
- 需要落库的关键对象必须能在文档中找到名称
- 重要字段需要固定时，必须写出字段名
- 枚举需要固定时，必须写出全部枚举值
- 业务编码、主键、租户字段、状态字段、类型字段等必须明确规则

## 13. API And Facade Rules

- 对外 `Facade` 必须单独列出
- 重要 `Facade` 必须明确输出内容
- 必要时必须明确固定方法名
- 必要时必须明确固定入参
- 必要时必须明确固定返回结构
- `DTO` 契约稳定性必须写入文档

## 14. Cache Rules

- 缓存必须单独成节
- 缓存键必须完整列出
- 缓存内容必须明确列出
- 缓存失效触发必须明确列出
- 必须明确缓存是否主动失效
- 必须明确 `TTL` 是主策略还是兜底策略
- 不允许只写“使用缓存”而不说明缓存形式

## 15. Security Rules

- 密码规则必须明确：
  - 哈希算法
  - `cost factor`
  - `salt` 策略
  - 明文禁止项
  - 重置密码规则
  - 首次登录修改密码规则
- 会话失效规则必须明确：
  - 用户停用
  - 用户删除
  - 管理员改密
  - 租户停用
- 登录方式如果有多种，必须说明统一用户主体与身份标识模型的关系

## 16. Audit Rules

- 审计日志必须单独定义
- 审计日志必须明确记录对象
- 审计日志必须明确记录字段
- 审计日志必须明确禁止记录的敏感内容
- 审计日志必须明确查询维度
- 审计日志必须明确失败处理策略

## 17. Import And Export Rules

- 导入导出如果在范围内，必须单独成节
- 必须明确支持的对象
- 必须明确文件格式
- 必须明确导入策略
- 必须明确模板字段
- 必须明确回执内容
- 必须明确是否支持更新、覆盖、合并

## 18. Open Item Rules

- 未确认内容必须集中写在 `Open Items`
- 已确认内容不得继续保留在 `Open Items`
- `Open Items` 为空时写 `无`

## 19. Output Quality Standard

文档应满足以下标准：

- AI 可直接读取并生成表结构设计
- AI 可直接读取并生成接口契约
- AI 可直接读取并生成服务和领域模型
- 工程师无需额外猜测关键规则
- 文档内部不存在前后冲突
- 文档内部不存在重复但不一致的描述
