# Bacon 项目故事

本文是 Bacon 的演进叙事，不是实现规范。

- 不作为工程规则依据
- 不作为 AI 默认输入文档
- 只记录“这支团队如何判断、取舍、修正”

## 序章：先把系统搭起来，也先把协作方式埋下伏笔（2026-03-19 ~ 2026-03-23）

最早一段提交几乎都是 `init`。  
这不是“没内容”，而是作者有意识地把风险压到最前面：先把仓库骨架、模块边界、启动路径搭起来，再谈细节优化。

这段时间里，作者做了两件后面反复出现的事：

1. 用最小可运行结构保证项目可生长。
2. 接受“先跑起来、再收口”的迭代节奏。

后面我们看到的大量小步治理，其实都能追溯到这个起点：  
先保证系统有稳定落点，再用连续小改把它变得可协作、可维护。

## 第一阶段：底座爆发，不先堆功能，先统一系统语言（2026-03-24）

3 月 24 日是第一次密集拐点（单日高密度提交）。  
从 short message 看，作者不是盲目铺功能，而是按“底座先行”推进：

1. 先补运行基础：`core/cache/web/security/swagger/boot`。
2. 再统一接口语法：响应包装、异常处理、参数校验、分层命名。
3. 最后才大规模补业务入口（UPMS/Auth/Inventory）。

代表性动作：

- `Feat(web): 统一业务接口响应包装`
- `Feat(web): 补充请求异常统一处理`
- `Feat(web): 接入参数校验能力`
- `Refactor(interfaces): 统一 provider 与 facade 包结构`
- `Docs(architecture): 固化对象命名规则`

这一阶段的核心判断很朴素但很关键：  
如果系统“说话方式”不统一，后续所有业务扩张都会变成返工。

## 第二阶段：生产语境接管，先做失败路径，再做漂亮路径（2026-03-26 ~ 2026-03-28）

项目从“有接口”进入“可在线”。  
作者集中处理库存、订单、支付、存储的失败路径和补偿链路。

代表性动作：

- `Feat(order): 下单链路接入Outbox执行与重试调度`
- `Feat(order): 幂等执行接入租约抢占并新增过期回收任务`
- `Feat(inventory): 增加审计outbox自动重试与死信机制`
- `Feat(inventory): 增加审计死信重放运营闭环`

作者的工程理念在这里非常清晰：

- 成功路径跑通只是起点。
- 线上稳定性由失败路径决定。

## 第三阶段：语义统一期，ID 与租户成为全仓“同一种语言”（2026-03-31 ~ 2026-04-06）

这段是全仓语义对齐：  
ID 强类型边界、tenant 语义、时间建模口径被系统收敛。

代表性动作：

- `Feat(common): 新增统一ID基础设施`
- `Refactor(upms): 使用TenantId替代TenantNo值对象`
- `Docs(architecture): 统一绝对时间点建模规则`
- `Refactor(ids): 收敛跨模块ID字段为Long`

期间出现回退与重做，这不是噪音，而是作者在坚持一个原则：  
先让跨域契约稳定，再继续推进内部重构。

## 第四阶段：规则工程化，经验从“知道”变成“阻断”（2026-04-07 ~ 2026-04-13）

这一阶段功能新增不是重点，可维护性才是重点。  
作者把隐性经验显式化，并压进工具链。

代表性动作：

- `Test(architecture): 新增实体构造器规则校验`
- `Docs(governance): 新增领域简单枚举统一写法规则`
- `Build(root): 接入覆盖率与静态分析质量门`
- `Chore(ci): 拆分PR质量门并去除重复执行`

方法论是稳定的：

- 文档先定口径
- 测试再做门禁
- 流程最终强制执行

## 第五阶段：从“收边界”到“收执行”，治理开始可连续迭代（2026-04-14 ~ 2026-04-18）

这一阶段表面主题很多：Facade、异常、request、Repository、ArchUnit、TODO、boot。  
但主线只有一条：从外到内收缩边界，再把收缩过程本身做成可执行系统。

### 5.1 先收跨域外层语言

先统一 Facade 契约和 provider 边界，避免跨域调用继续发散。

### 5.2 再收高风险语义

异常语义、请求模型放置、编号语义（如 `storedObjectNo`）被集中校准，减少协作噪音。

### 5.3 再收内部结构

Repository 命名、RepositoryImpl 与 PersistenceSupport 的边界被持续收紧，开始从“命名一致”走向“结构一致”。

### 5.4 最后收执行顺序

TODO 从“问题列表”变成“执行路由”：  
不只记录没做什么，还明确先做什么、为何先做、做完删什么。

## 第六阶段：最近 72 小时，AI 从编码助手进化为治理节拍器（2026-04-19 ~ 2026-04-21）

这一段提交密度很高，且呈现高频小步节奏。  
重点不是“提交多”，而是提交模式发生了变化：文档、门禁、修复、记录回收形成闭环。

### 6.1 upms 与跨域契约：快速试错、即时回收

`upms` 连续出现收口、回退、重提，说明作者在用小步可回滚方式压低试错风险。  
这不是反复横跳，而是“先验证再固化”的工程化节奏。

### 6.2 ArchUnit 门禁进入高频落地

最近 72 小时出现了一条非常清楚的序列：

- `Docs(governance): 新增RepositoryImpl单仓储实现规则`
- `Test(architecture): 增加RepositoryImpl单域单仓储门禁并清理TODO`
- `Test(architecture): 增加interfaces依赖白名单门禁并清理TODO`
- `Test(architecture): 增加IllegalArgumentException异常出口门禁并清理TODO`
- `Test(architecture): 扩展application装配门禁并清理TODO`
- `Docs(governance): 新增注解位置白名单规则`
- `Test(architecture): 扩展注解白名单与assembler装配门禁`

这条链路体现的不是“写了更多规则”，而是工程理念在执行层的落地：

1. 文档先定义边界。
2. 测试把边界变硬规则。
3. 模块失败立即修复。
4. TODO 只保留未完成项。

### 6.3 失败即修复：门禁触发后的四域小步闭环

架构门禁触发后，作者没有回避，而是按模块小步修复：

- `Fix(auth): 下沉认证DTO装配到application assembler`
- `Fix(inventory): 下沉预占条目DTO构造到assembler`
- `Fix(storage): 下沉分页结果DTO构造到application assembler`
- `Fix(payment): 下沉审计日志DTO构造到application assembler`

这组提交揭示了治理动作的“可运行版本”：

- 规则不是文档口号，而是能炸出真实问题。
- 问题不是挂账，而是马上以小步提交修掉。

### 6.4 TODO 与故事同步：记录系统开始自动维护

最近提交还出现了明显的记录治理动作：

- `Docs(todo): 调整执行顺序并删除空白分类`
- `Chore(governance): 同步TODO完成项并接入注解白名单门禁`
- `Chore(governance): 拆分TODO为模块化门禁任务`

这说明作者在尝试一种新的工作方式：  
让 AI 不只“写代码”，还参与自动提交和工作记录维护，把治理流程持续化。

## 贯穿全程的一条主线：AI 在变强，但方法先行

回看全程，前期大量 `init` 和最近几天的高密度提交，其实指向同一条演进线：

- 创世期：快速搭骨架，先保证系统可生长。
- 治理期：高频小步，先保证系统可协作。
- 当前：AI 承担执行节拍，人持续掌握边界判断与优先级。

作者的工程理念并没有因为 AI 介入而改变，反而更清晰：

1. 先判断主要风险。
2. 再排序动作顺序。
3. 用门禁固化共识。
4. 用小步提交控制偏差。
5. 用持续维护的记录系统确保团队与 AI 同步上下文。

所以 Bacon 的成长，不只是功能累积。  
更重要的是：它正在形成一套“AI 可持续执行、团队可持续接手”的工程方法。
