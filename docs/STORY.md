# Bacon 项目故事

本文是 Bacon 的演进叙事，不是实现规范。

- 不作为工程规则依据
- 不作为 AI 默认输入文档
- 只记录“这支团队如何判断、取舍、修正”

## 序章：创世期（2026-03-19 ~ 2026-03-23）

最早一段提交几乎都是 `init`。这不是“没内容”，而是作者在用最短路径把仓库骨架、模块边界、启动方式先搭起来。

这段的思路很朴素：

- 先把系统拆成可生长的多模块
- 先保证能起步，再进入规则化

在这一段尾声，出现了 `security: split current user provider by runtime mode`，说明作者很早就意识到“运行时模式差异”会是长期问题，先把入口分开，后面治理才不会互相牵连。

## 第一阶段：底座爆发（2026-03-24）

3 月 24 日是第一次密集拐点（单日 100+ 提交）。从 short message 看，作者当天不是盲目铺功能，而是按“底座先行”的顺序推进：

1. 先补运行与基础能力：`core/cache/web/security/swagger/boot`。
2. 再统一接口语法：响应包装、异常处理、参数校验、命名分层。
3. 最后大规模补业务入口：UPMS、Auth、Inventory 的核心接口闭环。

代表性动作包括：

- `Feat(web): 统一业务接口响应包装`
- `Feat(web): 补充请求异常统一处理`
- `Feat(web): 接入参数校验能力`
- `Refactor(interfaces): 统一 provider 与 facade 包结构`
- `Docs(architecture): 固化对象命名规则`

作者当时的判断很清楚：

- 如果先堆业务，后补底座，会造成全仓返工。
- 先让“系统怎么说话”统一，再让“系统说什么”扩张。

## 第二阶段：生产语境接管（2026-03-26 ~ 2026-03-28）

这一阶段，项目从“有接口”进入“可在线”。作者把精力集中在库存、订单、支付、存储四条主链路的失败路径与补偿路径。

### 2.1 库存与订单：先扛住失败，再追求漂亮

从提交可见连续动作：

- 订单接入 outbox、dead letter、幂等执行、租约抢占、过期回收
- 库存补齐审计 outbox、重放、死信、告警、并发重试
- 查询从内存过滤下推到仓储分页

代表性提交：

- `Feat(order): 下单链路接入Outbox执行与重试调度`
- `Feat(order): 幂等执行接入租约抢占并新增过期回收任务`
- `Feat(inventory): 增加审计outbox自动重试与死信机制`
- `Feat(inventory): 增加审计死信重放运营闭环`

思路是：

- 成功路径先跑通只是起点。
- 线上稳定性由失败路径决定。

### 2.2 存储域上线：从骨架到补偿闭环

3 月 27~28 日，`storage` 从模块初始化快速长成完整链路：

- 普通上传 + 分段上传
- 会话持久化 + 超时清理
- 审计出站 + 补偿重试 + 死信清理
- provider 鉴权、路径约束、契约测试

这说明作者没有把存储当“附件模块”，而是按生产域标准建设。

### 2.3 文档与规则同步推进

同一时期出现大量 `Docs(...)` 与 `Test(architecture)`，比如分层约束、命名约束、Facade 落位。

关键取舍：

- 文档先给口径
- 测试再变门禁

## 第三阶段：统一 ID 与租户语义（2026-03-31 ~ 2026-04-06）

这段是全仓“语义对齐期”。从 short message 可见，作者围绕两个高频冲突做了系统收敛：

1. ID 体系（强类型、Long/String 边界、发号策略）
2. 租户语义（tenantId/tenantNo、上下文透传、持久化字段）

代表性推进：

- `Feat(common): 新增统一ID基础设施`
- `Refactor(upms): 使用TenantId替代TenantNo值对象`
- `Docs(architecture): 统一绝对时间点建模规则`
- `Refactor(ids): 收敛跨模块ID字段为Long`

这期间出现过多次回退（Revert）和局部修正，反映了作者的真实决策过程：

- 先试一版全局统一
- 发现业务语义冲突后回退
- 再以“跨域契约稳定优先”重新落地

也正因为有这轮往返，后续模块协作的词义明显更稳定。

## 第四阶段：规则工程化（2026-04-07 ~ 2026-04-13）

这段不是新增功能高峰，而是“可维护性高峰”。作者把大量隐性经验显式化：

- 实体构造器规范
- 简单枚举统一写法
- ArchUnit 规则扩展
- Spotless / Checkstyle / CI 门禁
- 上下文透传与并发约束

代表性提交：

- `Test(architecture): 新增实体构造器规则校验`
- `Docs(governance): 新增领域简单枚举统一写法规则`
- `Build(root): 接入 Spotless 单一格式化工具`
- `Feat(ci): 新增PR质量门禁流水线`

作者在这一阶段的核心判断是：

- 规范如果只靠记忆，会在协作中衰减。
- 规范必须进入“工具链 + 测试 + 提交流程”。

## 第五阶段：边界收口开始深入实现（2026-04-14 ~ 2026-04-19）

这一阶段表面上看，提交主题很多：Facade、异常、request、Repository、ArchUnit、TODO、boot 测试。  
但把 short message 按时间线连起来看，作者其实只是在做同一件事：

- 先把跨域边界收口
- 再把这些边界压进实现
- 最后把治理过程本身变成可持续执行的系统

也就是说，这不是“很多零散重构”，而是一轮从外到内的收缩。

### 5.1 先统一跨域说话方式

这一段最先出现的是一串非常整齐的 Facade 契约提交：

- `Refactor(order): 收敛Facade查询契约`
- `Refactor(payment): 收敛Facade请求响应契约`
- `Refactor(upms): 收敛Facade请求响应契约`
- `Refactor(auth): 结构化Facade跨域契约`
- `Refactor(order): 统一Facade请求响应模型`

这里能看出作者的判断顺序：

- 如果跨域说话方式不统一，后面的任何模型治理都会不断反弹。
- 所以必须先把 `FacadeRequest / FacadeResponse`、查询契约、provider 接口边界这些“外层语言”收紧。

这时候作者关注的还不是某个领域内部怎么拆，而是系统各域之间如何以稳定方式互相调用。

### 5.2 再把文档口径变成测试门禁

随后很快跟上的是：

- `Docs(architecture): 明确Facade契约并补充ArchUnit用例`
- `Docs(governance): 增加api契约基础跨域依赖规则`
- `Test(common): 增加api跨域模块依赖架构检测`
- `Test(common): 增强架构规约检测`
- `Test(common-test): 拆分Path/Layer架构测试并对齐门面约束`

这一步很典型地体现了作者一贯的节奏：

- 文档先定语言
- 测试再把语言变成硬边界

作者已经不满足于“大家知道应该怎么做”，而是要求工程本身能拦住“做错了也照样通过”的情况。

### 5.3 然后处理高风险语义：异常、命名、请求模型

在 Facade 规则站稳之后，作者开始处理最容易污染协作语义的几类问题：

- `Fix(upms): 移除upms-api对upms-domain依赖`
- `Refactor(order): 应用层异常改为标准业务异常`
- `Refactor(upms): 应用层异常语义化分类`
- `Refactor(upms): 仓储实现改用NotFound与BadRequest异常`
- `Docs(governance): 补充异常使用约定`
- `Refactor(*): 请求模型迁移到interfaces.request`（auth/inventory/order/payment/storage/upms）
- `Refactor(storage): 存储对象契约切换storedObjectNo并同步upms头像引用`

这几类动作的共同点是：

- 它们都直接影响“别人如何理解这个系统”
- 一旦词义不稳，AI、测试、跨模块调用都会反复踩坑

所以作者在这里优先止血的，不是代码长短，而是语义噪音：

- 异常要能表达业务判断，而不是落回 `IllegalArgumentException`
- 请求模型要停留在 interfaces，不再四处漂浮
- `storedObjectNo` 这样的业务编号必须说清楚，不能和“主键 id”混在一起

### 5.4 TODO 开始从记录问题变成安排顺序

同一时期 `todo` 相关提交很密：

- `Chore(todo): 同步控制器规则落地状态与治理清单`
- `Chore(todo): 跨域扫描后重整统一治理清单`
- `Docs(todo): 补充ArchUnit可落地待办`
- `Docs(todo): 清理已完成治理项`
- `Docs(todo): 更新治理任务顺序与现状`

这背后其实是一个很重要的转变：

- TODO 不再只是“还没做什么”
- 而开始承担“下一步先做什么、为什么先做这个”的职责

作者显然已经意识到，随着治理项越来越多，如果不显式安排顺序，团队和 AI 都会在“每件事都对，但不知道先做哪件”里打转。

### 5.5 到了 4 月 18 日，治理重心从外层契约转向内层实现

最近 48 小时的提交最有意思的地方在这里。

前一阶段作者主要在收口“边界怎么说”，到了这两天，他开始深入到这些边界背后的实现层：

- `Refactor(domain): 统一Repository命名白名单`
- `Refactor(domain): 去除Repository的upsert语义`
- `Refactor(domain): 收敛Repository短命名`
- `Refactor(domain): 统一收敛领域仓储命名`
- `Refactor(infra): 收敛仓储与持久化支撑命名`
- `Test(arch): 新增仓储实现依赖硬规则`

这说明作者的思路又往前走了一步：

- 光把接口名字说顺，不够。
- 还要让 Repository、RepositoryImpl、PersistenceSupport 这些内部结构本身也服从同一套语言。

于是治理顺序变成了：

1. 先给 Repository 方法命名设白名单。
2. 去掉最模糊的 `upsert` 语义。
3. 再把长名字、业务味名字、数据库味名字压回统一词汇表。
4. 最后把这套约束交给 ArchUnit。

这里能看出作者非常稳定的偏好：

- 先收词汇表
- 再收结构表
- 最后才谈具体拆分

因为如果语言没稳，任何拆分都只是把混乱复制得更碎。

### 5.6 upms：不是急着拆小，而是先变得可拆

这一轮里，`upms` 是最集中的实验场：

- `Feat(upms-domain): 统一编码值对象与领域规则`
- `Refactor(upms-domain): 统一租户编码字段命名`
- `Refactor(upms): 收敛infra仓储边界`
- `Refactor(upms): 拆分仓储实现与持久化支持`
- `Refactor(upms): 适配拆分后的应用层仓储契约`
- `Refactor(upms): 对齐关系仓储 support 命名`

如果只看标题，容易觉得作者是在“继续整理 upms”。  
但把顺序连起来看，会发现这其实是在做拆分前置条件：

1. 先把 code、id、字段名这些词汇统一。
2. 再把主仓储、关系仓储、support 依赖边界拉开。
3. 再让 application 和 infra 对上新的仓储契约。

也就是说，作者没有急着去拆 `UserApplicationService` 或 `UserController`，因为他知道：

- 内部词汇不稳，拆分只会复制混乱。
- 仓储边界不稳，application 再怎么分，也还是会被 infra 的业务味实现拖回去。

所以 `upms` 这轮工作的真实方向不是“马上变小”，而是“先变得可拆、可控、可继续收缩”。

### 5.7 order：开始把旁路事实从主单本体旁边拿开

`order` 的轨迹也很完整：

- `Refactor(order): 使用值对象收敛仓储契约`
- `Refactor(order-domain): 收拢订单领域建模`
- `Refactor(order-application): 收拢订单用例编排`
- `Refactor(order-infra): 收拢订单仓储实现`
- `Refactor(order-interfaces): 收拢订单接口契约`
- `Refactor(order): 拆分订单派生仓储并补齐测试`

这一串动作背后的判断也很清楚：

- 支付快照、库存快照、审计日志虽然都围着订单转，但它们不是“主单本体”。
- 如果长期挂在主仓储下面，主仓储会越来越像一个什么都装的大抽屉。

所以作者选择的不是“继续忍”，而是把这些派生事实单独拿出来，让仓储重新表达数据的真实生命周期和职责边界。

这很能反映作者的一个稳定倾向：

- 聚合相关，不等于仓储必须合并。
- “放一起方便”如果没有克制，最后通常会变成“没人敢动”。

### 5.8 application 与 boot：治理开始补最后一环

到了最近几条提交，作者又把注意力放到了一个很容易被忽略的地方：

- `Refactor(application): 补充事务边界标注`
- `Refactor(boot): 收敛装配测试上下文`
- `Test(boot): 补齐仓储拆分后的测试桩`

这里的信号很明确：

- 结构和命名治理如果不回到事务边界、boot 装配、测试桩，最后就只是“代码看起来更干净”。
- 真正的闭环，是实现层、装配层、测试层都跟着新的结构一起稳定下来。

尤其是事务这一条，很能看出作者并不把 `@Transactional` 当装饰：

- 不是“所有写方法都贴上”
- 而是逐域判断哪些入口该由 Spring 托管事务，哪些流程已经由显式事务执行器控制

这说明作者现在已经从“治理结构”进一步走到“治理边界是否真实可运行”。

## 这一段变化最重要的，不是重构数量，而是思路顺序

把 2026-04-14 到 2026-04-19 连起来看，作者的心路历程其实非常完整：

1. 先收外层契约，避免跨域沟通继续发散。
2. 再把这些契约变成规则和门禁。
3. 然后处理最容易污染语义的异常、命名、请求模型。
4. 接着把 TODO 从记录清单改造成执行顺序。
5. 最后把治理深入到 domain / repository / infra / application / boot 的真实接缝。

所以这段故事真正值得记录的，不是“最近做了很多整理”。

而是作者已经越来越稳定地采用一套可重复的方法：

- 先收语言
- 再收边界
- 再收门禁
- 最后收执行顺序

## 尾声：这条思路是如何稳定下来的

回看全部 short message，作者的思路并不是线性完美，而是持续迭代的：

- 先搭骨架，再扩功能
- 先统一语言，再放大协作
- 先处理高风险边界，再做风格统一
- 先写规则，再把规则做成门禁
- 出现偏差就小步回退，不把整批改动带偏

所以 Bacon 的成长，不只是一堆功能累积。

更重要的是，这套节奏已经能看见作者稳定的思考方式：

- 先判断主要风险在哪
- 再决定动作顺序
- 出现偏差就小步纠偏
- 最终把经验沉淀成规则与门禁

这让项目的进化不再依赖“某次冲刺”，而是依赖“可重复的决策方法”。
