# Bacon 项目编年史

本文是项目消遣文档，用于回望开发过程。

- 本文不是工程规则
- 本文不是实现依据
- 本文不作为 AI 输入文档

## 序章：地基初立（2026-03-19 至 2026-03-21）

项目以连续的 `init` 和一次 `Initial commit` 开场。

这几天的提交信息几乎没有业务语义，更多是在反复确认仓库骨架、模块边界和初始工程形态。它看起来安静，实际并不轻松。一个多模块系统的第一批提交，往往决定后面所有工作的展开姿势。Bacon 的起点并不华丽，但方向很明确：先把地基圈出来，再让业务落进去。

## 第一纪：规则先于规模（2026-03-23）

`2026-03-23` 是项目第一次明显脱离“初始化状态”的一天。

这一天的关键动作包括：

- 按运行模式拆分当前用户提供器
- 校正安全链路中的运行时优先级
- 校验微服务模式下当前用户提供器装配
- 忽略 `.DS_Store`
- 明确每次文件修改后都要提交
- 统一 AI 协作相关的提交消息格式

从这批提交可以看出，项目的第一轮推进不是直接铺业务，而是先建立秩序。工程约束、提交纪律和运行模式边界在早期就被拉直，说明开发者从一开始就把“后续是否可控”看得很重。

## 第二纪：基础设施成形（2026-03-24）

`2026-03-24` 是第一波密集建设日。

基础设施层在这一天迅速铺开：

- 全局 `Jackson` 配置
- `JetCache` 和 `Redis` 配置
- `RestTemplate` 与 `RestClient`
- `WebMvc` 配置
- 通用异常体系
- 验证码服务
- `Swagger` 文档分组和输出
- 统一响应包装
- 参数校验与请求异常统一处理

与此同时，认证与权限链路也开始闭合：

- 增加 `OAuth2` 安全链配置
- 增加密码登录验证码与 `RSA` 流程
- 放开 `Swagger` 文档访问
- 统一认证接口路径前缀和响应包装

UPMS 也在同一天进入核心能力补齐期，用户、角色、部门、租户、菜单、岗位、资源、系统访问日志等能力开始成片出现。

这一天之后，项目第一次显出完整后端平台的轮廓：它不再只是模块集合，而是开始拥有统一的接口语言、统一的异常语义和统一的装配边界。

## 第三纪：库存进入工业化阶段（2026-03-25 至 2026-03-26）

接下来的重心迅速转向库存域。

这一阶段的提交不再满足于“接口可用”，而是明显转向“生产可用”：

- 接入 `TinyId`
- 接入订单与支付业务单号生成器
- 增加 `Seata` 自动配置
- 为库存远程调用增加韧性、降级、熔断和告警
- 统一 `BizException` 与错误码体系
- 修复库存仓储 Bean 缺失、回退条件和单体装配问题
- 默认启用严格仓储，仅保留显式内存降级
- 统一分页参数归一化并增加统一上限

真正让这一阶段变重的，是库存审计与补偿链路的落地：

- 新增审计 `outbox` 落库
- 增加自动重试与死信机制
- 增加死信重放和运营闭环
- 增加指标、告警规则模板、DDL、种子数据
- 补充关键分层、并发和集成测试

这说明库存已经不是“先写个 CRUD”的层次，而是进入了异常语义、并发控制、失败补偿、重试边界和运行告警一并建模的阶段。

## 第四纪：支付闭环与存储开疆（2026-03-27 至 2026-03-29）

`2026-03-27` 是全仓又一个高压建设日。

支付域在这一天完成了一次完整闭环：

- 完善支付需求与数据库基线
- 补全支付领域与应用层
- 新增支付持久化仓储实现
- 增加远程调用、回调请求、仓储兜底、回退和集成测试
- 收紧支付失败语义和审计边界
- 补齐支付审计查询子域

同一天，存储域几乎平地起楼：

- 初始化存储模块骨架
- 建立存储领域模型、持久化、本地装配和远程门面
- 新增普通上传与分段上传契约
- 接入基于 `S3API` 的对象存储封装
- 打通上传会话持久化与本地分段上传完成链路
- 增加尺寸限制、统一校验、超时清理、两阶段删除
- 新增存储审计日志、补偿出站、补偿重试、死信清理
- 补齐分段上传 `DDL`、上线整改清单和接口编排规则

`2026-03-29`，认证域继续推进：

- 新增认证仓储模式与 `OAuth` 客户端持久化
- 新增 `Redis` 认证会话与授权仓储
- 修复严格仓储默认装配条件

从这几天开始，项目的业务版图基本成形，Auth、UPMS、Inventory、Payment、Storage 都开始有了自己的完整链路。

## 第五纪：统一标识与生产持久化转向（2026-03-31）

`2026-03-31` 的主线非常清楚：统一 ID，退出开发期仓储模式。

这一天里，多个动作彼此呼应：

- 新增统一 ID 基础设施
- 推进 `UserId`、`TenantId`、部门标识等统一建模
- 统一对外契约中的 `tenantId`
- 统一绝对时间点为 `Instant`
- 收敛租户主数据模型与租户编号边界

同时，项目显式切换到生产基线：

- 将各业务域内存仓储迁入测试代码
- 移除仓储模式配置
- 推进 `MyBatis` 持久化成为默认实现
- 完善仓储与缓存生产规则文档

这一阶段意味着工程态度的变化：过去为了快速起步而存在的临时装配和内存实现，开始主动让位于稳定的生产路径。

## 第六纪：强类型化与架构工程化（2026-04-01 至 2026-04-03）

从 `2026-04-01` 开始，Bacon 进入“全面收口”的阶段。

一方面，UPMS 和多个业务域持续推进强类型与稳定模型：

- `PostId`、`MenuId`、`ResourceId`、`UserCredentialId` 等主键建模
- 用户身份、凭据状态、岗位状态、资源状态、角色类型、数据范围等切换为领域枚举
- 收紧租户编码规则，增加稳定编码与过期时间
- 补齐头像存储、权限缓存和租户相关边界

另一方面，工程治理从文档升级为可执行规则：

- 新增命名与目录规则
- 新增 ArchUnit 基类和分层依赖校验
- 限制 `SysLog` 与 `Transactional` 使用位置
- 统一 `Mapper`、`RepositoryImpl`、`DO`、`persistence` 等目录规则
- 清理已完成的架构待办

同一时期，订单域经历了非常密集的一轮重构：

- 收敛订单主键、用户标识、租户标识
- 收敛金额、币种、状态、分页查询条件
- 收敛库存快照、支付快照、死信记录、出站事件、审计日志模型
- 明确值对象与枚举边界
- 让订单号生成器直接返回值对象

这意味着项目不再满足于“字段能传通”，而是在逼领域模型具备可维护的语义密度。

## 第七纪：数值 ID 收口与战略回撤（2026-04-04 至 2026-04-05）

`2026-04-04` 到 `2026-04-05` 的关键词是：统一，但不盲目。

先是统一 ID 能力继续增强：

- 增加链式发号兜底
- 增加雪花兜底开关
- 重构发号配置模型
- 抽取长整型 ID 基类

随后，多个域开始大规模向数值 ID 收口：

- 部门、菜单、岗位、角色、用户、资源等主键切换到 `Long` 或强类型数值 ID
- Facade 主键契约全面跟进
- 订单主键改为数值 ID
- 存储对象 ID 同步调整
- 支付、库存、租户等边界持续统一

但就在统一推进最猛烈的时候，项目也留下了一次非常关键的回撤：

- 审计人字段 Long 化后被连续回滚
- 对应测试与文档同步回滚
- 随后明确治理规则：审计人标识统一为字符串

这组提交说明，项目在推进一致性时并没有牺牲真实边界。当统一方案与业务语义冲突时，开发者选择修正路线，而不是强行保留表面整齐。

同一天，UPMS 仍在继续前进：

- 新增 Department 排序字段与接口
- 用户导入改用部门编码
- 收敛状态枚举分层与查询契约

回撤没有让项目停下，只是让路线更准确。

## 第八纪：边界打磨与规则封口（2026-04-06 至 2026-04-07）

最后两天，提交重心从大块建设转入细部打磨。

主要动作包括：

- 适配头像更新强类型对象 ID
- 统一菜单与部门根节点空值语义
- 收敛 Auth、UPMS、Order、Payment 中租户与主键的数值边界
- 修复 Long 类型收口后的一批测试和参数错误
- 简化订单用户、租户、出站事件主键取值
- 统一库存审计事件标识模型、仓库标识值对象与死信回放状态
- 删除仅测试使用或仅兼容字符串语义的多余构造
- 新增实体构造器规则校验

这是一种很典型的封边阶段：不再扩张系统面积，而是持续清理多余语义、统一入口、消灭含混模型，让架构约束逐渐变成代码自然呈现出的样子。

## 第九纪：枚举归一与构造器定型（2026-04-08）

`2026-04-08` 的节奏非常集中，几乎整天都在做一件事：把“已经说清楚的规则”继续压进代码形态。

这一天首先完成了简单枚举写法的统一闭环：

- 新增领域简单枚举统一写法规则
- 修改枚举命名规则
- 统一简单枚举的字符串转换风格
- 为 `common-test`、`order`、`inventory` 增加简单枚举架构校验

这不是一次单点重构，而是把前几天形成的工程约束继续从文档推进到测试，再从测试推进到各业务域实现。到这里，枚举已经不再只是“值能对上就行”，而是连 `from` 方法、命名方式和字符串转换口径都开始趋于单一。

另一条并行主线，是实体构造器规范真正进入多域收口期：

- 优化实体边界构造器规则提示
- 收敛实体构造器架构规则
- `Auth`、`UPMS`、`Storage` 增加实体与枚举架构用例
- `Auth`、`UPMS`、`Storage` 分别收敛实体构造器与状态、标识模型
- `Order` 继续删除私有构造器、修正边界构造器委托并统一辅助方法

这说明项目开始明确拒绝“为了兼容历史写法而保留额外构造入口”。构造器的数量、参数顺序、委托关系和可见性都被持续压缩，目标不是让实体更灵活，而是让实体更难被错误创建。

库存域则在这一天继续承担“规则试验田”的角色：

- 审计出站、回放任务、库存流水、预占单与明细继续值对象化
- 多个状态字段改为枚举，移除字符串常量
- 引入释放原因枚举
- 修正测试仓储中的租户与主键比较
- 提取 DTO 组装复用逻辑并清理多余取值调用

库存的这批提交延续了它一贯的气质：不仅要把模型做强类型化，还要顺手把测试、组装器、仓储比较逻辑一起收干净，不给“半新半旧”的中间态留下太多空间。

同一天，UPMS、Auth、Storage 与基础设施也都在继续小步收束：

- 消除租户应用服务事务自调用
- 让部门查询提供者直接接收 `Long` 主键
- 收敛远程存储门面构造器
- 清理仓储与查询辅助转换
- 修复全仓 `checkstyle` 阻塞项

如果说前几天的重点是“把规则说出来”，那这一天更像是在做最后一层压实。规则没有新增太多抽象名词，但它们开始系统性地体现在枚举、构造器、测试和辅助代码的每一个细部上。

## 第十纪：编解码归位与测试总攻（2026-04-09）

`2026-04-09` 是一次非常典型的“边修边证实”的工程推进日。

这一天最先被拉直的，是库存域接口和应用层之间的边界：

- 收敛库存接口中的状态字符串边界
- 修正 Provider 契约测试桩签名
- 修正审计任务与内存仓储中的类型转换
- 固定审计补偿分页状态边界

这批动作看起来分散，实际上目标很一致：把 DTO、字符串、值对象、枚举和应用服务签名之间的职责重新摆正。库存域继续承担规则试验田的角色，但这一次不只是改模型，也把测试入口和异常语义一起校正了。

随后，工程治理又向前推了一步：

- 明确架构规则默认保留 `.because()`
- 调整 `DisplayName` 与 `.because()` 的文档口径
- 收敛接口层依赖与状态参数校验
- 收敛应用层编解码命名规则

这里最重要的变化，不是单个类名，而是语义分工终于被说得更清楚了。`Assembler`、`Codec`、`Mapper` 不再只是“喜欢怎么叫就怎么叫”的风格问题，而是开始被文档、目录和架构测试同时约束。命名从审美问题，变成了可以执行的结构规则。

真正把这一天推向高潮的，是两笔很小、但极具代表性的提交：

- `Refactor(auth): 收敛令牌编解码器目录归属`
- `Refactor(order): 收敛出站载荷编解码器目录归属`

它们说明新规则并没有停在库存域内部，而是开始反向检查整个系统。当 `Codec` 规则进入全仓校验后，Auth 和 Order 中遗留的“旧位置但新语义”类被逐个揪出，再以很克制的小步提交方式收回正确目录。这不是大重构，却非常能体现项目当下的气质：允许规则晚一点落地，但一旦规则成立，就会认真把全仓补齐。

最后，这一天以一次真正意义上的总验证收尾：

- 全局执行 `mvn clean test`
- 持续修复跨域规则外溢点
- 最终打通全部 61 个模块

这一步让前面所有“小修小补”真正获得了证明。很多项目在规则重构阶段会停留在“局部能过”，而 Bacon 在这一天继续推进到“全仓都过”。这不是写下更多规范，而是用一次完整构建告诉自己：这些规则已经不只是文档意图，它们已经成为系统当前能够承受的现实。

## 尾声

从 `2026-03-19` 到 `2026-04-09`，Bacon 的演进轨迹非常鲜明：

- 前期快速搭建骨架
- 中期完成基础设施、安全、认证和统一接口体系
- 随后补齐库存、支付、存储等复杂业务域的失败语义和补偿链路
- 再之后推进统一 ID、统一命名、统一目录、统一事务与持久化边界
- 最终把治理规则从文档落到测试、构造器和目录结构中

它不是一条单纯追求交付速度的开发曲线，而是一条边建设、边纠偏、边治理、边工程化的曲线。

对这个项目来说，真正重要的不只是“功能做出来了”，而是它逐渐具备了长期维护、持续演进和抵抗混乱的能力。
