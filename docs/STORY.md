# Bacon 项目故事

本文结合现有故事稿与最近 48 小时内的 git commit short message 重新整理。

- 本文不是工程规则
- 本文不是实现依据
- 本文不作为 AI 默认输入文档

## 开场

Bacon 一开始并没有急着铺业务。

它先把仓库骨架、运行模式、模块分层、协作纪律这些看起来不热闹、但以后一定会反复碰到的东西先钉住。启动方式怎么切，接口怎么装，提交怎么写，哪些文档给人看、哪些规则给机器判，这些地基先被压实，后面的业务推进才没有一路返工。

所以 Bacon 的起点，不是“先做个能跑的系统”，而是“先把一群会继续长大的系统，放进能长期收口的框架里”。

## 第一幕：从能跑，到像个系统

地基打稳之后，平台层和通用能力很快成形。

统一响应、参数校验、异常处理、Swagger、Web 配置、缓存、Redis、HTTP 客户端、验证码、安全链路，这些能力被一口气铺开。与此同时，Auth 和 UPMS 也开始长出第一批真正可用的主链路：登录、用户、角色、部门、菜单、租户、资源、岗位、系统日志。

这一阶段最重要的变化，不是“接口数量变多了”，而是 Bacon 开始拥有统一语言：

- 接口长什么样，越来越像同一家系统写出来的
- 错误怎么抛，越来越可预期
- 模块怎么依赖，越来越有边界感

项目从这里开始，不再是模块列表，而是有秩序地说同一种话。

## 第二幕：业务开始进入生产语境

真正把 Bacon 推进到另一个阶段的，是库存、订单、支付、存储这几条链路先后进入“生产语境”。

库存不再只关心扣减成功，而是开始关心审计、补偿、重试、死信、回放、并发争抢、租约抢占和告警指标。订单不再只是下单动作本身，而是开始认真处理跨域编排、幂等、超时回收、快照和查询下推。支付补上回调、审计与失败边界。存储则从上传接口一路长到分段上传、对象会话、补偿出站和清理任务。

这时候的 Bacon，明显开始像一个会在线上长期运行的系统，而不是一个用例演示工程。

它开始正视失败：

- 请求会失败
- 任务会重跑
- 消息会积压
- 数据会冲突
- 补偿会变成日常，而不是例外

也正因为这一轮推进，Inventory、Order、Payment、Storage 几个域逐渐都长出了自己的筋骨。

## 第三幕：统一语言，收回边界

再往后，项目进入了一段很典型的“重写内部语法”的时期。

统一 ID、值对象、状态枚举、时间模型、仓储主路径、分层命名、目录规则、ArchUnit 约束，这些事看起来不像新功能，但它们在悄悄改变项目的可维护性上限。

这段时间里，Bacon 做了几件关键的事：

1. 统一绝对时间到 `Instant`
2. 推进强类型 ID 和业务值对象
3. 把内存仓储逐步赶回测试和演示位置
4. 让 `MyBatis` 和数据库重新成为真相源
5. 把“文档规定”逐渐变成“测试会拦”的架构规则

更重要的是，Bacon 不是一味做“大一统”，而是在不断试错之后，慢慢学会把东西放回它真正应该待的地方。

公共模块不再什么都收，真正共享的才留下；领域自己的 ID、自己的语义、自己的状态，开始回到各自领域。统一不再意味着“都塞进 common”，而是意味着“边界清楚，转换清楚，职责清楚”。

这是 Bacon 开始变成熟的标志。

## 第四幕：租户从显式参数，退回上下文

最近这段时间，故事的主线非常清楚：租户透传正在从显式参数，一步步退回到上下文能力。

最先动起来的是 Inventory、Order、Payment。然后是 Auth、UPMS、common 和测试体系一起跟进。最近 48 小时的提交，几乎都围着这条线在收口：

- `Refactor(inventory):` 持续收敛透传、持久化组装、实体创建入口
- `Refactor(order):` 删除查询链路租户参数，前移主键生成，清理薄封装
- `Refactor(payment):` 删除领域实体 `tenantId` 字段，收敛持久化映射
- `Refactor(common):` 新增上下文执行助手，统一 ID 转换命名为 `Codec`
- `Build(common):` 接入 checkstyle，并明确和 spotless 的职责边界
- `Fix(common):` 修复存量 checkstyle 违规
- `Docs(order):` 补充订单、库存、支付补偿链路文档
- `Refactor(upms):` 收口租户透传与对象组装
- `Refactor(auth):` 收口登录与改密租户透传
- `Refactor(common):` 收紧上下文透传与分层约束

这条线背后的判断，其实很朴素：

如果一个租户是“当前上下文”，那它就不应该在 application、domain、interfaces 里被到处传递。  
如果一个 ID 是“业务对象本身”，那它才应该继续留在方法签名里。

于是 Bacon 开始做几件成套出现的事：

- 删除 DTO 里的 `tenantId`
- 删除 facade 和 service 里无意义的 `tenantId` 参数
- 让 infra 从 `BaconContextHolder` 取当前租户
- 让 MQ、provider、remote facade、HTTP client 自动带上下文头
- 让 `create(...)` 与 `reconstruct(...)` 的职责重新分明
- 在 application 新增 assembler / codec，明确 DTO、值对象、领域对象之间的转换位置

这一轮改动表面上在“删参数”，实际上是在把一件长期让代码发虚的事讲清楚：

租户是运行上下文，不是到处搬运的包袱。

## 第五幕：规则不再靠记忆，而是靠门禁

最近 48 小时还有另一条很重要的暗线：规则开始真正落地成门禁。

项目不再满足于“大家知道 spotless 负责格式、checkstyle 负责规约”，而是把这件事写进治理文档、接进工作流、接进 `mvn test`，再用代码修掉第一轮暴露出来的存量问题。

同样的事情也发生在架构分层上：

- `create()` 只给 application
- `reconstruct()` 只给 infra
- `@SysLog`、`@Transactional` 的位置继续被收紧
- 分层依赖方向不再只靠约定，而是靠 ArchUnit 直接拦

这意味着 Bacon 正在从“靠开发者自觉维持整洁”，走向“靠规则把整洁变成默认结果”。

这是一个项目从年轻走向稳健时，非常关键的一步。

## 尾声

如果用一句话概括 Bacon 到目前为止的变化，大概是：

它先学会把系统做出来，再学会把系统收紧，最后开始学会让正确的做法不再依赖记忆。

最近两天的提交尤其像一场安静但重要的收官动作：

- 文档把原则说得更少，但更准
- 代码把参数传得更少，但语义更清楚
- 架构把自由留得更少，但边界更稳定

故事并没有结束。

只是 Bacon 现在越来越不像一个“正在堆功能的仓库”，而像一套已经开始形成自己语法、自己纪律、自己节奏的系统。
