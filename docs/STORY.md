# Bacon 项目故事

本文结合现有故事稿与最近一批 git commit short message 重新整理。

- 本文不是工程规则
- 本文不是实现依据
- 本文不作为 AI 默认输入文档

## 开场

Bacon 一开始并没有急着铺业务。

它先把仓库骨架、运行模式、模块分层、协作纪律这些看起来不热闹、但以后一定会反复碰到的东西先钉住。启动方式怎么切，接口怎么装，提交怎么写，哪些文档给人看、哪些规则给机器判，这些地基先被压实，后面的业务推进才没有一路返工。

所以 Bacon 的起点，不是“先做个能跑的系统”，而是“先把一群会继续长大的系统，放进能长期收口的框架里”。

## 第一幕：从能跑，到像个系统

地基打稳之后，平台层和通用能力很快成形。

统一响应、参数校验、异常处理、Swagger、Web 配置、缓存、Redis、HTTP 客户端、验证码、安全链路，这些能力被一口气铺开。与此同时，Auth 和 UPMS 也开始长出第一批真正可用的主链路：登录、用户、角色、部门、菜单、租户、资源、岗位、系统日志。

这一阶段最重要的变化，不是“接口数量变多了”，而是 Bacon 开始拥有统一语言：

- 接口长什么样，越来越像同一家系统写出来的
- 错误怎么抛，越来越可预期
- 模块怎么依赖，越来越有边界感

项目从这里开始，不再是模块列表，而是有秩序地说同一种话。

## 第二幕：业务开始进入生产语境

真正把 Bacon 推进到另一个阶段的，是库存、订单、支付、存储这几条链路先后进入“生产语境”。

库存不再只关心扣减成功，而是开始关心审计、补偿、重试、死信、回放、并发争抢、租约抢占和告警指标。订单不再只是下单动作本身，而是开始认真处理跨域编排、幂等、超时回收、快照和查询下推。支付补上回调、审计与失败边界。存储则从上传接口一路长到分段上传、对象会话、补偿出站和清理任务。

这时候的 Bacon，明显开始像一个会在线上长期运行的系统，而不是一个用例演示工程。

它开始正视失败：

- 请求会失败
- 任务会重跑
- 消息会积压
- 数据会冲突
- 补偿会变成日常，而不是例外

也正因为这一轮推进，Inventory、Order、Payment、Storage 几个域逐渐都长出了自己的筋骨。

## 第三幕：统一语言，收回边界

再往后，项目进入了一段很典型的“重写内部语法”的时期。

统一 ID、值对象、状态枚举、时间模型、仓储主路径、分层命名、目录规则、ArchUnit 约束，这些事看起来不像新功能，但它们在悄悄改变项目的可维护性上限。

这段时间里，Bacon 做了几件关键的事：

1. 统一绝对时间到 `Instant`
2. 推进强类型 ID 和业务值对象
3. 把内存仓储逐步赶回测试和演示位置
4. 让 `MyBatis` 和数据库重新成为真相源
5. 把“文档规定”逐渐变成“测试会拦”的架构规则

更重要的是，Bacon 不是一味做“大一统”，而是在不断试错之后，慢慢学会把东西放回它真正应该待的地方。

公共模块不再什么都收，真正共享的才留下；领域自己的 ID、自己的语义、自己的状态，开始回到各自领域。统一不再意味着“都塞进 common”，而是意味着“边界清楚，转换清楚，职责清楚”。

这是 Bacon 开始变成熟的标志。

## 第四幕：上下文退到幕后，契约走到台前

最近这段时间，故事的主线已经不只是“减少租户透传”了。更准确地说，是系统开始把一些曾经被混在一起的东西，慢慢分开。

最早动起来的是 `Auth`、`Inventory`、`Order`、`Payment`、`Storage` 这些真正要穿过请求、编排、持久化的链路。`Refactor(auth): 收口登录与改密租户透传`、`Refactor(inventory): 收口透传与对象组装`、`Refactor(order): 收口透传与持久化组装`、`Refactor(payment): 收口透传与持久化组装`、`Refactor(storage): 收口对象组装与审计透传`，看起来都只是局部收口，但一连着看，就能感觉到一种取舍正在重复出现：有些信息开始往后退，有些信息开始被留在真正该说话的地方。

于是后面的提交沿着这条线继续收紧。`Refactor(common): 收紧上下文透传与分层约束` 先把公共语义收住，`Refactor(upms): 收口租户透传与对象组装` 再把组织域一并拉回同一条轨道。再往后，`Refactor(inventory): 删除过薄租户上下文包装` 这样的提交，已经不太在意“看起来完整”，而更在意留下来的那一层，到底还剩不剩必要性。

但 Bacon 并没有停在“少传几个参数”。

很快，另一组提交把焦点转向了契约本身。`Refactor(order): 收紧应用层值对象契约`、`Refactor(order): 统一仓储分页参数语义`、`Refactor(order): 统一订单分页查询入口`，把订单域里原本略有松动的查询边界重新对齐。`Refactor(storage): 统一分页参数契约` 与 `Refactor(storage): 收紧存储对象分页查询入参边界`，让存储域的分页语义从入口到仓储都变得更像一件完整的东西。`Refactor(payment): 提取支付订单查询装配器`、`Refactor(upms): 提取用户身份与数据范围装配器` 则把那些原本散在流程里的拼接动作，慢慢安放回各自的位置。

这一段代码看久了，会发现它的变化并不张扬。没有大开大合的改名，也没有刻意制造新的概念。它更像是在一层层把松的地方拧紧，把含糊的地方说清。等这些提交连成一片时，系统里哪些是路过的，哪些是该留下的，也就慢慢显出来了。

## 第五幕：统一并不靠口号，而是靠持续校准

如果说上一幕是在收回上下文，那么这一幕就是在校准系统说话的方式。

`Refactor(web): 统一非法枚举请求语义` 看起来只碰了错误处理，但它让入口上的许多细碎分歧，少了一层摇摆。`Fix(upms): 调整部门与菜单参数校验分层`、`Fix(upms): 校准仓储实现契约`、`Refactor(upms): 明确仓储持久化语义并收敛查询枚举`，则继续把 UPMS 里那些“能跑，但说得不够准”的边角往回修。

随后，提交开始更明显地围绕“接口边界”展开。`Refactor(upms): 统一分页与接口参数契约`、`Refactor(upms): 收敛菜单部门日志应用层边界`、`Refactor(upms): 统一用户租户应用层入参边界`、`Refactor(upms): 收敛资源角色租户用户接口边界`，这些名字读起来像一连串局部整理，但它们合起来，像是在把一座已经盖起来的房子重新校门框。门还在，路也还在，只是人走过去时，不再总要侧一下身。

这也解释了为什么文档和测试会在同一时间被不断补强。`Docs(architecture): 明确接口值对象与分页契约边界` 给出书面口径，`Test(architecture): 增加应用层协议边界规则`、`Test(common): 增加DTO映射架构约束`、`Test(common): 收敛架构规则格式`、`Test(common): 调整架构规则校验格式` 则把这些口径安静地接到门口。以后再有人从这里经过，系统会先替大家记住一次。

## 第六幕：路径、注册与部署，开始服务于全局秩序

最近这一批提交里，还有一条不那么喧哗、但很说明阶段变化的线：系统开始把“怎么被找到、怎么被接入、怎么被部署”也纳入同一套秩序里。

`Refactor(interfaces): 按domain规范整改公开与provider路径`、`Refactor(gateway): 统一公开路由与OAuth2域前缀`、`Docs(governance): 补充微服务URL路径命名约束`、`Test(architecture): 增加URL前缀ArchUnit约束`，让 URL 这件事终于不再只是“能通就行”。路径开始像字段名、像包名一样，被认真对待。它们不再只是通往功能的门牌，也顺手把系统内部的分寸暴露了出来。

而 `Feat(register): 引入Nacos注册发现并移除冗余启动模块`、`Feat(deploy): 新增双部署样例并接入微服务注册发现`，则把这种秩序延伸到运行方式本身。到这里，Bacon 没有急着把架构写成口号，而是先让节点如何被发现、应用如何被启动、部署样例如何落在仓库里，彼此能对得上。很多时候，系统真正的形状，不是在图里先出现的，而是在这些细部长期不打架之后，才慢慢露出来。

## 第七幕：文档开始记录差异，而不是掩盖差异

故事走到现在，连文档的性格也变了。

`Docs(todo): 补充模块结构差异清单` 与 `Docs(todo): 删除待办文件` 这两条提交放在一起看，很能说明 Bacon 现在的文档气质。它不再把待办单当成一种温和的拖延，而是更愿意先承认差异还在，再把已经失效的悬念收掉。`Docs(readme): 整理仓库入口说明` 也是同一脉络的一部分：入口说明不求热闹，只求少一点误会。

到这里，文档也开始像代码一样，有了取舍。该说的说准，不该留的删掉。表面上只是几份文档在整理，实际更像项目终于接受了一件事：真实状态并不总是整齐的，但只要描述得足够准确，它就能继续往前长。

## 尾声

如果用一句话概括 Bacon 到目前为止的变化，大概是：

它先学会把系统做出来，再学会把系统收紧，最后开始学会让正确的做法不再依赖记忆。

最近这一批提交尤其像一场安静但重要的收官动作：

- 文档开始更诚实地记录差异，也更果断地删除失效前提
- 代码把上下文藏回幕后，把契约推到真正该出现的位置
- 架构不再急着扩张能力，而是在接口、路径、分页、装配这些细处反复校准
- 规则从说明书变成门禁，系统的秩序感开始有了执行力

故事并没有结束。

只是 Bacon 现在越来越不像一个“正在堆功能的仓库”，而像一套已经开始形成自己语法、自己纪律、自己节奏的系统。
