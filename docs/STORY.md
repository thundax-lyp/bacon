# Bacon 项目编年史

本文结合现有编年史与 git commit message 重新整理，用于回望项目推进过程。

- 本文不是工程规则
- 本文不是实现依据
- 本文不作为 AI 输入文档

## 序章：地基初立（2026-03-19 至 2026-03-23）

项目以连续的 `init` 和一次 `Initial commit` 开场。

这一阶段的提交几乎没有业务语义，重点是把仓库骨架、模块层次和启动形态搭起来。直到 `2026-03-23`，项目才第一次从“初始化噪声”中显出明确方向：

- 按运行模式拆分当前用户提供器
- 校正运行时优先级
- 校验微服务模式下的装配链路
- 忽略 `.DS_Store`
- 明确每次文件修改后都要提交
- 统一 AI 协作提交消息格式

这说明 Bacon 的第一步不是急着铺业务，而是先把协作纪律、运行边界和工程秩序拉直。项目从一开始就把“后面能不能持续收敛”看得很重。

## 第一纪：平台骨架与接口秩序（2026-03-24）

`2026-03-24` 是第一次真正意义上的平台建设日。

基础设施层在这一天迅速成形：

- 全局 `Jackson` 配置
- `JetCache`、`Redis`、`RestTemplate`、`RestClient`
- `WebMvc` 配置与统一异常体系
- 验证码服务
- `Swagger` 文档分组与标签整理
- 统一响应包装与参数校验

认证链路也在同一天闭合出第一版轮廓：

- 增加 `OAuth2` 安全链配置
- 增加密码登录验证码与 `RSA` 流程
- 放开 `Swagger` 文档访问
- 统一认证接口路径前缀与响应语义

UPMS 则开始批量补齐核心管理能力：

- 用户、角色、部门、租户、菜单、岗位、资源管理接口
- 系统访问日志查询
- 角色授权回显

这一天之后，Bacon 不再只是模块集合，而是开始拥有统一接口语言、统一异常语义和统一装配边界。

## 第二纪：库存工业化起势（2026-03-25 至 2026-03-26）

随后重心迅速转向库存域，而且不是停留在“接口能跑”的层次，而是直接奔着生产语义去。

核心动作包括：

- 接入 `TinyId`
- 接入订单与支付业务单号生成器
- 增加 `Seata` 自动配置
- 强化库存远程调用韧性、参数校验、熔断与告警
- 统一 `BizException` 与错误码体系
- 修复严格仓储装配与回退条件
- 默认启用严格仓储，仅保留显式内存降级
- 统一分页参数归一化与上限

真正让库存进入工业化阶段的，是审计与补偿链路的落地：

- 审计 `outbox` 落库
- 自动重试、死信与异步重放
- 租约抢占、`CAS` 重试与并发退避
- 指标、告警规则模板、`DDL`、种子数据
- 关键分层、并发和集成测试

同一时期，订单域也开始补齐生产能力：

- 数据库仓储主实现落地
- 下单链路接入 `Outbox` 与重试调度
- 接入库存、支付跨域编排
- 引入命令幂等、租约抢占与过期回收
- 分页查询下推到仓储层

这意味着项目开始认真处理失败语义、并发边界、仓储主路径和线上运行风险，而不再满足于演示级闭环。

## 第三纪：支付闭环与存储开疆（2026-03-27 至 2026-03-29）

`2026-03-27` 是又一个高密度建设日。

支付域在这一天完成闭环：

- 完善支付需求与数据库基线
- 补全支付领域、应用层与持久化仓储
- 增加回调请求、远程调用、回退与集成测试
- 收紧支付异常、失败审计与查询边界
- 补齐支付审计查询子域

存储域也在同一天几乎平地起楼：

- 初始化存储模块骨架
- 建立领域模型、持久化、本地装配和远程门面
- 拆分普通上传与分段上传契约
- 接入基于 `S3API` 的对象存储封装
- 打通上传会话持久化与本地分段上传完成链路
- 增加尺寸限制、统一校验、超时清理、两阶段删除
- 增加审计日志、补偿出站、补偿重试和死信清理

与此同时，项目还在做一件很容易被忽视、但对长期维护很重要的事：固化对象注释、字段顺序、目录归属和分层命名规则。业务在扩张，结构也在同步收束。

到了 `2026-03-29`，认证域继续补齐生产持久化：

- 增加认证仓储模式与 `OAuth` 客户端持久化
- 新增 `Redis` 认证会话与授权仓储
- 修复严格仓储默认装配条件

从这几天开始，Auth、UPMS、Order、Inventory、Payment、Storage 都已经拥有较完整的主链路。

## 第四纪：统一 ID 与生产持久化转向（2026-03-31）

`2026-03-31` 的主线非常明确：统一标识，退出开发期仓储模式。

关键动作包括：

- 新增统一 ID 基础设施
- 推进 `UserId`、`TenantId`、部门标识等统一建模
- 对外契约中的 `tenantId` 逐步统一
- 统一绝对时间为 `Instant`
- 收敛租户主数据模型与租户编号边界

同时，项目显式切向生产基线：

- 各业务域内存仓储迁入测试代码
- 移除仓储模式配置
- 推进 `MyBatis` 持久化成为默认实现
- 补充仓储与缓存生产规则文档

这一天代表一种态度转换：临时加速手段开始主动让位于稳定的生产路径。

## 第五纪：强类型化与架构工程化（2026-04-01 至 2026-04-03）

从 `2026-04-01` 开始，Bacon 明显进入“全面收口”阶段。

一方面，多个业务域持续推进强类型建模：

- `PostId`、`MenuId`、`ResourceId`、`UserCredentialId` 等主键建模
- 用户身份、凭据状态、岗位状态、资源状态、角色类型、数据范围等切换为领域枚举
- 收紧租户编码规则，补齐稳定编码与过期时间
- 收敛头像存储、权限缓存和租户边界

另一方面，工程治理开始从文档升级为可执行规则：

- 新增命名与目录规则
- 增加 ArchUnit 基类和分层依赖校验
- 限制 `SysLog` 与 `Transactional` 使用位置
- 统一 `Mapper`、`RepositoryImpl`、`DO`、`persistence` 等目录语义

订单域在这一阶段完成了一轮高强度重构：

- 收敛订单主键、用户标识、租户标识
- 收敛金额、币种、状态和分页查询条件
- 收敛库存快照、支付快照、死信记录、出站事件和审计日志模型
- 让订单号生成器直接返回值对象

项目不再满足于“字段能传通”，而是在逼领域模型具备稳定且可维护的语义密度。

## 第六纪：数值 ID 收口与战略回撤（2026-04-04 至 2026-04-05）

这两天的关键词是：统一，但不盲目。

统一 ID 能力继续推进：

- 增加链式发号兜底
- 增加雪花兜底开关
- 重构发号配置模型
- 抽取长整型 ID 基类

随后，多域开始大规模向数值 ID 收口：

- 部门、菜单、岗位、角色、用户、资源等主键切换到 `Long` 或强类型数值 ID
- `Facade` 主键契约全面跟进
- 订单主键改为数值 ID
- 存储对象 ID 同步调整
- 支付、库存、租户等边界持续统一

但项目也在这时留下了一次很关键的回撤：

- 审计人字段 Long 化后被连续回滚
- 对应测试与文档同步回滚
- 随后明确治理规则：审计人标识统一为字符串

这组提交说明，项目推进一致性时并没有牺牲真实业务边界。统一方案一旦与语义冲突，就会立刻修正路线。

## 第七纪：边界打磨与规则封口（2026-04-06 至 2026-04-08）

从 `2026-04-06` 开始，项目重点从大块建设转入细部封边。

这一阶段的关键变化包括：

- 适配头像更新强类型对象 ID
- 统一菜单与部门根节点空值语义
- 收敛跨模块 ID 字段为 `Long`
- 修复 Long 类型收口后的测试与参数错误
- 简化订单用户、租户、出站事件主键取值
- 统一库存审计事件标识模型、仓库编号值对象和死信回放状态
- 删除只为兼容旧字符串语义而保留的多余构造

`2026-04-08` 的主题尤其集中，几乎都在把“已经说清楚的规则”继续压进代码形态：

- 新增领域简单枚举统一写法规则
- 修改枚举命名规则
- 统一简单枚举 `from` 与字符串转换风格
- 为 `common-test`、`order`、`inventory` 增加简单枚举架构校验
- 优化实体边界构造器规则提示
- `Auth`、`UPMS`、`Storage` 增加实体与枚举架构用例
- `Inventory` 持续推进值对象化与状态枚举化

如果说前几天是在“把规则说出来”，那这一阶段更像是在把规则压实到构造器、枚举、测试和辅助代码的每一个细节里。

## 第八纪：编解码归位与全仓验证（2026-04-09）

`2026-04-09` 是一次非常典型的“边修边证实”的推进日。

库存域先被拉直的是接口层与应用层之间的类型边界：

- 收敛状态字符串边界
- 修正 `Provider` 契约测试桩签名
- 修正审计任务与内存仓储中的类型转换
- 固定审计补偿分页状态边界
- 收敛库存实体创建与重建入口
- 审计日志、死信、库存主键改由应用层生成强类型标识

工程治理则继续前推：

- 接入 Spotless 单一格式化工具
- 增加实体创建与重建分层约束
- 明确架构规则默认保留 `.because()`
- 收敛应用层编解码命名规则

更有代表性的，是一批“把标识归还给所属域”的提交：

- `OrderId` 收回订单域
- `PaymentOrderId` 收回支付域
- 组织权限标识收回 UPMS 域
- 身份与凭证标识收回认证域
- `SkuId` 开始向 `common-commerce` 迁移试点

这些动作说明系统不再追求“所有公共对象都放公共模块”，而是更重视边界所有权。公共能力开始只保留真正共享、稳定且低争议的部分。

## 第九纪：上下文透传与租户边界隐身化（2026-04-10）

`2026-04-10` 的主题非常鲜明：把租户透传从显式参数，逐步收敛为上下文能力。

主线动作包括：

- 新增上下文基础对象
- 切换租户上下文透传主链路
- 新增异步上下文透传能力
- 新增租户消息头透传
- 为库存新增租户自动隔离
- 为 `MyBatis` 增加租户注解开关语义

基于这条主线，多个跨域接口同步收口：

- 收敛跨域租户透传
- 删除跨域门面租户参数
- 删除库存实体中的租户字段
- 收敛库存门面租户透传与编译边界

这是一种很有代表性的成熟化信号。早期系统常常把租户信息层层显式下传，虽然直观，却容易造成参数污染、职责扩散和漏传风险。到这一天，Bacon 开始把租户视为真正的运行时上下文，而不是每层都要手工搬运的业务入参。

更值得注意的是，这条路线并没有线性前进：

- 上下文基础对象曾被回滚后重做
- `MyBatis` 租户注解写入语义也出现过一次回撤
- 与之配套的库存改造清单文档同样经历了新增、回滚、再收敛

这些往返说明项目不是在机械推进“大一统方案”，而是在一边试探最合适的边界，一边用小步回滚控制风险。最终留下来的方向很明确：租户隔离要更底层、更自动，也更少侵入业务模型。

同一天，治理继续补强：

- 新增并发使用约束规则
- 调整库存审计实体租户字段顺序
- 继续收敛库存领域模型值对象

这使得 `2026-04-10` 不只是一次租户改造日，更像是项目从“强类型治理”进一步迈向“运行时上下文治理”的起点。

## 尾声

从 `2026-03-19` 到 `2026-04-10`，Bacon 的演进轨迹已经非常清晰：

- 先搭仓库骨架与协作纪律
- 再建立统一接口、异常、安全与基础设施
- 随后补齐库存、订单、支付、存储等复杂业务闭环
- 再推进统一 ID、强类型模型、目录规则与架构测试
- 最后开始收敛上下文透传、租户隔离和跨域参数边界

它不是一条单纯追求交付速度的曲线，而是一条边建设、边纠偏、边治理、边工程化的曲线。

对这个项目来说，真正重要的不只是“功能做出来了”，而是它逐渐具备了长期维护、持续演进和抵抗混乱的能力。
